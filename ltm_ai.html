Spring Data (13:48:21)

4. What mechanisms are there for pagination in Spring Data?

Spring Data provides excellent support for pagination through the Pageable interface and the PagingAndSortingRepository.

  • PageRequest: You use PageRequest.of(pageNumber, pageSize, Sort) to create a Pageable object, which is then passed to the repository method.
  • Return Types:
    • Page<T>: Returns a page of entities along with pagination metadata (total pages, total elements). This executes an extra COUNT query.
    • Slice<T>: Returns a page of entities and a boolean indicating if there is a next slice. It avoids the expensive COUNT query, making it more efficient for large datasets (like infinite scrolling).
    • List<T>: Just returns the elements without any metadata.
  • Offset vs. Keyset Pagination: While standard Spring Data uses offset-based pagination (which gets slower for deep pages), keyset (or seek) pagination can be implemented using custom queries or libraries like Blaze-Persistence for better performance on massive datasets.

Core Java: Thread Safety (13:50:41)

5. How do you ensure thread safety in Java applications?

There are several strategies to ensure thread safety in Java, depending on the use case:

  • Immutability: The best way to avoid concurrency issues is to make objects immutable (using final fields, returning copies). Immutable objects are inherently thread-safe.
  • Statelessness: Designing classes (like Spring Singletons) to be stateless, meaning they don't hold any conversational state between method invocations.
  • Synchronization: Using the synchronized keyword for methods or blocks to enforce mutual exclusion, though this can introduce contention.
  • Locks: Utilizing explicit locks from java.util.concurrent.locks, such as ReentrantLock or ReadWriteLock, which provide finer control over locking mechanisms.
  • Concurrent Collections: Using collections from java.util.concurrent like ConcurrentHashMap, CopyOnWriteArrayList, or BlockingQueue instead of standard or synchronized collections.
  • Atomic Variables: Using classes like AtomicInteger or AtomicReference for lock-free, thread-safe operations on single variables using Compare-And-Swap (CAS).
  • ThreadLocal: Using ThreadLocal to give each thread its own isolated instance of an object.

Spring Boot Configuration (13:52:51)

6. What is the role of the application.properties or application.yml file?

These files are the primary mechanism for externalizing configuration in a Spring Boot application.

  • Environment Independence: They allow you to define properties (like database credentials, server ports, third-party API keys) outside the application code, enabling the same code to run in different environments (dev, test, prod).
  • Spring Boot Auto-configuration: Spring Boot uses these properties to automatically configure beans. For example, setting spring.datasource.url will trigger the auto-configuration of a DataSource bean.
  • Profiles: You can create profile-specific files (e.g., application-dev.yml, application-prod.yml) to activate different configurations based on the active Spring profile.
  • Custom Properties: You can define your own custom properties and bind them to Java objects using the @ConfigurationProperties or inject them directly using @Value.

Microservices: Security (13:54:43)

7. How can microservices handle authentication and authorization across services?

In a microservices architecture, security is typically handled centrally but enforced locally:

  • API Gateway as the Entry Point: The API Gateway often handles the initial authentication (e.g., terminating OAuth2/OIDC flows, validating JWTs) and routes requests to downstream services.
  • Stateless JWT (JSON Web Tokens):
    • The Identity Provider (e.g., Keycloak, Auth0) issues a JWT containing user claims and roles upon successful login.
    • The API Gateway passes this JWT to downstream microservices in the Authorization header.
    • Each microservice independently verifies the JWT's signature (using a shared public key) and extracts the roles to enforce authorization (e.g., using Spring Security's @PreAuthorize). This makes the system stateless and highly scalable.
  • Service-to-Service Authentication: For internal communication between microservices, mTLS (Mutual TLS) is often used to ensure encryption in transit and verify the identity of the calling service. Alternatively, services can use client credentials grant type to obtain tokens for internal calls.

Microservices: Resilience (13:56:50)

8. Scenario: A downstream microservice depends on a database that is down. How would you ensure resilience?

To prevent a database failure from causing cascading failures across the system, several resilience patterns can be applied:

  • Circuit Breaker Pattern: Use a library like Resilience4j. If the database calls fail repeatedly, the circuit breaker trips and immediately returns an error or a fallback response for subsequent calls, preventing resource exhaustion (like thread pool starvation) in the calling service.
  • Fallback Mechanism: When the circuit is open or a call fails, provide a graceful degradation path. This could mean returning cached data, default values, or simply a friendly "Service Unavailable" message instead of a generic 500 error.
  • Retries with Exponential Backoff: For transient network glitches, implement automatic retries. Crucially, use exponential backoff and jitter to avoid overwhelming the database as it tries to recover.
  • Timeouts: Always configure strict timeouts for database connections and queries to ensure threads aren't blocked indefinitely waiting for a response from a hanging database.
  • Bulkheads: Isolate resources (like thread pools) for database calls so that a database failure only exhausts its dedicated pool, leaving the rest of the microservice functional to serve other requests.

Coding Assessment (14:02:39)

10. Given an array of prices, where prices represents the price of a stock on day i, determine the maximum profit you can achieve. You may complete as many buy-sell transactions as you wish, but you must sell the stock before you buy again and you can hold at most one share at a time. If no profit can be made, return zero. Output the maximum total profit as an integer.

Approach: Greedy Algorithm

Since we can make as many transactions as we want, we can simply capture all upward price movements. We iterate through the array, and whenever the price on day i is greater than the price on day i-1, we add the difference to our total profit. This effectively captures the profit from every local minimum to every local maximum.

Code (Java):

class Solution {
    public int maxProfit(int[] prices) {
        int maxProfit = 0;
        for (int i = 1; i < prices.length; i++) {
            if (prices[i] > prices[i - 1]) {
                maxProfit += prices[i] - prices[i - 1];
            }
        }
        return maxProfit;
    }
}

Complexity Analysis:

  • Time Complexity: O(N), where N is the length of the prices array, because we iterate through the array exactly once.
  • Space Complexity: O(1), as we only use a single variable maxProfit to keep track of the accumulated profit, requiring constant extra space.
