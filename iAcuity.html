Microservices Architecture

1. What are the advantages of using microservices?
  • Independent Deployment & Scaling: Services can be deployed independently without redeploying the entire application. High-traffic services can be scaled horizontally without paying the cost to scale idle components.
  • Fault Isolation: If one service crashes (e.g., due to an out-of-memory error), it does not bring down the entire system.
  • Technology Agnostic: Different teams can choose the best language, database, or tech stack for their specific service.
  • Smaller Codebases: Easier for developers to understand, maintain, and test compared to massive monoliths.
2. What is an API Gateway, and how does it know which service to route requests to?

An API Gateway acts as the single entry point for all client requests. It knows how to route traffic by looking at the request metadata (URL path, HTTP headers, or query parameters) and matching it against a set of configured routing rules. In dynamic environments, the gateway talks to a Service Discovery registry to resolve the current IP addresses and ports of the target services before routing the request.

3. What is Service Discovery, and how do services register with it?

Service Discovery (like Netflix Eureka or HashiCorp Consul) is a central registry that keeps track of the network locations of all available microservices.

  • Registration: When a microservice instance starts up, it sends a network call to the registry declaring its IP address, port, and service name.
  • Maintenance: The service continuously sends "heartbeats" to the registry to prove it is healthy. If the registry misses several heartbeats, it deregisters the instance so traffic is no longer routed to it.

Database Design & Data Modeling

4. If you need to assign multiple roles to users, how many tables do you need, and how do you link them?

To handle a many-to-many relationship between Users and Roles, you need three tables:

  1. users: Stores user data. Primary key is user_id.
  2. roles: Stores role definitions (e.g., ADMIN, USER). Primary key is role_id.
  3. user_roles (Join/Mapping table): Contains two foreign key columns (user_id and role_id) that link back to the primary keys of the first two tables. A composite primary key made of both columns prevents duplicate assignments.

Application Logic, ORM, & Concurrency

5. What is the @PrePersist annotation, and why would you use it?

@PrePersist is a JPA (Java Persistence API) lifecycle callback annotation. It is attached to a method inside an entity class and is executed automatically right before the entity is inserted into the database for the first time.

  • Use cases: Automatically setting a createdAt timestamp, generating a UUID, or computing a custom username based on other entity fields before the INSERT statement fires.
6. If two concurrent requests try to create the same custom username at the same time, what happens? What occurs if we use the SERIALIZABLE isolation level?
  • Default Behavior: Without proper locks, a race condition occurs. Both threads might read the database, see the username is available, and attempt to insert it. If there is a unique constraint on the username column, the database will throw a UniqueConstraintViolationException on the second insert. If there is no constraint, duplicate usernames will be created.
  • Using SERIALIZABLE: SERIALIZABLE is the strictest database transaction isolation level. It prevents phantom reads by acquiring range locks or table-level locks. The transactions are forced to execute sequentially. The first transaction will succeed, and the second will either block until the first finishes (and then fail validation) or immediately throw a serialization failure exception. While safe, this heavily degrades application performance and throughput.

Performance & Optimization

7. After a user is created, the email-sending logic takes a lot of time. How can we optimize this using an asynchronous model?

You should decouple the slow email-sending process from the main HTTP request thread so the user doesn't experience a delayed response.

  • Message Queues (Best Practice): The API saves the user to the database, publishes a UserCreatedEvent message to a broker (like Kafka, RabbitMQ, or AWS SQS), and immediately returns a 200 OK to the client. A separate worker service consumes that message and sends the email in the background.
  • Thread Pools / @Async: In a Spring Boot app, you can annotate the email method with @Async. The main web thread hands the task off to a background thread pool and immediately returns the HTTP response.
